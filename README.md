# admit

Three fixed-capacity cache eviction policies in Go: LRU, LFU, and W-TinyLFU. Zero
dependencies, stdlib only. Built to answer one question with numbers instead of
assumptions: does admission control actually beat plain LRU, and where does it fall
short of plain LFU?

Answer: yes, and on stale workloads. Details below.

## The problem with the default

Most caches reach for LRU without thinking about it. LRU has a specific failure mode:
it's purely recency-based, so it can't tell a genuinely hot key from a key that was
merely touched once, recently. A single sequential scan (a batch job, a full-table
export, a crawler) evicts your entire hot working set, because every scanned key looks
"most recently used" even though none of them will be touched again.

LFU fixes that: it tracks actual access frequency, so one-hit-wonders can't displace
proven-hot keys. But naive LFU never forgets. A key that was hot last week keeps its
frequency count forever and stays pinned in cache, starving whatever's hot now. It also
cold-starts badly, a brand new key needs to survive with frequency 1 against entries
with frequency 1000.

W-TinyLFU (Einziger & Friedman, 2015) is the fix Caffeine and Ristretto both ship as
their default: keep a small LRU window to absorb one-hit-wonders and give new keys a
fighting chance, and gate everything that leaves the window through an approximate,
*aging* frequency filter before it's allowed into the main cache. Aging is the part
plain LFU is missing.

## Design

**`Cache[K, V]`** (`cache.go`) is the shared interface. All three policies implement
it with O(1) amortized `Get`/`Set`, backed by an intrusive doubly linked list shared
across implementations rather than three copies of the same list code.

**Count-Min Sketch** (`sketch.go`) is the frequency estimator behind admission. 4-bit
counters, packed 16 to a `uint64` word, so the whole sketch costs about half a byte per
counter instead of 8 bytes for a naive `int64` slice. A key's four counter probes come
from one 64-bit hash run through four different avalanche mixes, not four separate hash
functions, one multiply-mix per probe is cheap; three extra full hash computations
aren't. Collisions only ever inflate an estimate, never deflate it, which is the right
failure mode here: a false admit wastes one cache slot, a false reject costs nothing.

The table is sized to roughly 16x the cache's capacity in counters (matching Caffeine's
actual sizing, not a guess), because every `Get` and `Set` touches the sketch for its
key whether or not that key is resident. Cardinality flowing through the sketch is much
larger than the cache itself, and a table sized to the raw capacity collapses into noise
under any real load. Getting this wrong quietly destroys the whole admission filter,
worth calling out because it's exactly what happened during development here: the first
version sized the table to capacity 1:1, and W-TinyLFU couldn't beat LRU on the scan
workload until it was fixed.

**Aging**: every ~10 additions per counter, the whole table is halved in place using a
bit trick (`(word >> 1) & 0x7777...`) that halves all 16 nibbles in one shift instead of
16 separate masked shifts. This is what makes the sketch track *recent* frequency
instead of all-time frequency, and it's the one thing plain LFU can't do.

**W-TinyLFU** (`wtinylfu.go`): a 1% LRU window plus a Segmented LRU main cache split
80/20 into protected/probation. New keys always enter the window. When the window
overflows, its LRU victim becomes an admission candidate: if there's free room in main
it's admitted for free (still warming up), otherwise it has to beat the estimated
frequency of main's current probation victim to get in at all. A key earns protected
status, and real eviction immunity, on its second access. Ties favor the incumbent, on
purpose, to avoid thrashing on noise.

**Baselines** (`lru.go`, `lfu.go`) exist so the comparison isn't against a strawman:
both are proper O(1) implementations, not something deliberately hobbled to make
W-TinyLFU look good.


Three workloads, three different stories:

- **Stationary Zipfian**: W-TinyLFU tracks LFU almost exactly, both clearly ahead of
  LRU. Expected, a well-tuned frequency filter should converge on LFU's behavior when
  the distribution never changes.
- **Scan-polluted** (hit ratio measured only over the repeatable hot keys, since the
  scan keys are uncacheable by construction and would just dilute the number for every
  policy equally): LRU loses 5 to 10 points of hit ratio to the scan. LFU and
  W-TinyLFU barely notice.
- **Concept drift** (hot key set flips halfway through, hit ratio measured only on the
  second half): this is where LFU falls apart, worse than LRU at every cache size, and
  catastrophically worse at small ones (28.58% vs LRU's 77.70% at 1%). Its frequency
  counts from the first half never decay, so it keeps defending a working set that no
  longer exists. W-TinyLFU's aging sketch recovers almost completely. LRU, having no
  memory at all, adapts fastest of the three here, the one workload where forgetting
  everything instantly is an advantage, and it's the reason W-TinyLFU keeps a window
  instead of routing 100% of traffic through the frequency filter.

Net: W-TinyLFU is the only policy that doesn't have a workload where it loses badly.
That's the actual argument for admission control, not "it's faster" (it isn't, see
below) but that it doesn't have a catastrophic failure mode.

Throughput, `go test -bench . -benchmem` (i3-1315U):

```
BenchmarkLRU-8        9,853,168   120.2 ns/op   48 B/op   1 allocs/op
BenchmarkLFU-8        9,228,642   129.8 ns/op   48 B/op   1 allocs/op
BenchmarkWTinyLFU-8   7,329,252   157.5 ns/op   48 B/op   1 allocs/op
```

W-TinyLFU costs about 30% more per operation than LRU, entirely the sketch probes.
That's the trade you're making: worse constant factor, no catastrophic failure mode.

## Usage

```go
c := admit.NewWTinyLFU[string, []byte](10_000, admit.HashString)

c.Set("key", []byte("value"))
v, ok := c.Get("key")
```

Keys are hashed by a function you provide, not by reflection. `HashString` covers
string keys; for anything else, write a `func(K) uint64` (an identity cast is fine for
integer keys, `Estimate`'s probes already do the avalanche mixing internally).

## Running it

```
go build ./...
go vet ./...
go test ./...
go test -bench . -benchmem
go run ./cmd/bench
```

Go 1.22+, no other setup.

## Limitations

- Not concurrent. No mutex, no sharding, no read-buffer batching. Wrap it yourself if
  you need that; adding it here wouldn't have changed any of the numbers above, and it
  would have buried the part of this project that's actually about eviction policy.
- No TTL, no size-weighted eviction. Every entry counts as one unit of capacity.
- Frequency is approximate by design (that's the whole point of a sketch), so
  admission decisions near a tie are noisy. Ties favor the incumbent.
- Keyed by an explicit hash function, not reflection over `K`. Deliberate: no
  `unsafe`, no `reflect`, and you don't get a silently bad hash for a type this package
  never tested against.
